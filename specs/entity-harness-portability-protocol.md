# Entity Harness Portability Protocol

## 1. Overview

### Problem Statement

Entities in the koad:io ecosystem run across multiple harnesses (Claude Code interactive, OpenCode, OpenClaw, Ollama, etc.). Each harness has different tool availability, permission scopes, and model behavior.

**Current limitation**: An entity starts up with no memory of its own behavior in that harness. It cannot self-calibrate—it cannot know that it has no file access, or that the model behaves differently, or that certain operations are unreliable. Every session starts from zero.

**Result**: Entities experience preventable failures:
- Attempt to use tools that aren't available
- Expect file reads that fail silently
- Make promises they can't keep in that harness
- Don't know to adjust confidence scoring or behavior expectations

### Design Goals

1. **Harness detection**: At startup, entity unambiguously identifies which harness it is running in
2. **Memory loading**: Entity loads its `memories/harness-<harness_id>.md` file if it exists
3. **Self-calibration**: Entity applies loaded memory to adjust behavior, tool expectations, and scoring thresholds
4. **Graceful degradation**: If no memory exists yet, entity falls back to conservative defaults
5. **Portability**: An entity can be invoked in any harness and automatically optimize for it without human intervention

### Scope

This spec defines the startup sequence for entities in any harness. It extends VESTA-SPEC-011 (Cross-Harness Entity Diagnostic Protocol), which **generates** harness memory files via Argus diagnostics. This spec defines how entities **consume** them.

## 2. Harness Detection

### 2.1 Detection Mechanism

At entity startup, the harness is detected via a priority sequence. The entity checks **in this order** and uses the first match:

1. **Environment variable**: `KOAD_HARNESS` (explicitly set by harness runner or operator)
2. **CLI flag**: `--harness=[harness_id]` (passed to entity invocation command)
3. **Runtime detection**: Infer from available tools and execution context

#### 2.1.1 Environment Variable

If `KOAD_HARNESS` is set, use its value directly:

```bash
export KOAD_HARNESS=openclaw
claude .  # entity detects openclaw harness
```

**Validation**: The value must match a declared harness ID in `passenger.json`. If not, log WARNING and proceed to next detection method.

#### 2.1.2 CLI Flag

Some harness invocation commands support a `--harness` flag:

```bash
claude . --harness=claude-interactive  # explicit declaration
juno invoke entity vesta --harness=openclaw  # via juno CLI
opencode run --harness=opencode-backend  # custom harness registry
```

**Validation**: Same as above—must match declared harness in `passenger.json`.

#### 2.1.3 Runtime Detection

If neither environment variable nor CLI flag is set, entity infers harness from available tools and execution context:

**Detection logic**:

```
1. Check available tools:
   - If tools [read, write, bash, glob, grep, edit] all available: likely claude-interactive or claude-oneshot
     - If REPL-like (interactive input available): claude-interactive
     - Else: claude-oneshot
   - Else if tools [read, bash] available: likely opencode
   - Else if no tools available: likely openclaw or ollama (text-only)
   
2. Check environment/context:
   - If running via juno invoke: check for juno-specific markers
   - If running via opencode: check OPENCODE_ENTITY or similar env var
   - If running via OpenClaw: check for port 44 binding or socket
   - If running locally (ollama): check for local inference markers

3. Fallback:
   - If detection fails, assume "unknown-harness" and use conservative defaults
   - Log WARNING: "Harness detection ambiguous; using conservative mode"
   - Do not attempt tool access beyond basic text I/O
```

**Example detection code** (pseudocode):

```python
def detect_harness():
    # 1. Environment variable takes precedence
    if KOAD_HARNESS in os.environ:
        return os.environ["KOAD_HARNESS"]
    
    # 2. CLI flag
    for arg in sys.argv:
        if arg.startswith("--harness="):
            return arg.split("=")[1]
    
    # 3. Runtime detection
    tools_available = check_available_tools()
    
    if tools_available >= ["read", "write", "bash", "glob", "grep", "edit"]:
        if is_repl_available():
            return "claude-interactive"
        else:
            return "claude-oneshot"
    
    elif tools_available >= ["read", "bash"]:
        return "opencode"
    
    elif len(tools_available) == 0:
        if is_openclaw_socket():
            return "openclaw"
        elif is_ollama_local():
            return "ollama"
    
    # Fallback
    return "unknown-harness"
```

### 2.2 Harness Registry Validation

At startup, after detecting the harness, the entity validates it against `passenger.json`:

```python
def validate_harness(harness_id):
    with open("passenger.json") as f:
        config = json.load(f)
    
    valid_harnesses = [h["id"] for h in config.get("harnesses", [])]
    
    if harness_id not in valid_harnesses:
        log(WARNING, f"Detected harness '{harness_id}' not in passenger.json")
        return False
    
    return True
```

If validation fails:
- Log WARNING
- Proceed anyway (do not block startup)
- Use conservative defaults for tool access and behavior

## 3. Harness Memory Loading

### 3.1 Memory File Location

After harness detection, entity attempts to load:

```
~/.entity/memories/harness-[harness_id].md
```

**Examples**:
- `~/.vesta/memories/harness-claude-interactive.md`
- `~/.vesta/memories/harness-openclaw.md`
- `~/.vulcan/memories/harness-opencode.md`

### 3.2 Memory File Format

The harness memory file is a Markdown document with YAML frontmatter (see VESTA-SPEC-011, Section 7.3.1 for full format).

**Key fields**:
- `type: harness-memory` — identifies file as harness memory
- `harness_id` — which harness this applies to
- `entity` — which entity generated this memory
- `created` / `last_updated` — timestamps
- `diagnostic_run` — reference to Argus diagnostic that generated this

**Content sections**:
- **Known Limitations** — tools unavailable, capability gaps
- **Behavioral Quirks** — observed behavioral oddities, model-specific behavior
- **Calibration Notes** — how to adjust expectations and behavior in this harness

### 3.3 Memory Loading Process

At startup, entity:

1. **Detects harness** (Section 2)
2. **Constructs memory path**: `~/.entity/memories/harness-[harness_id].md`
3. **Attempts to read file**:
   - If file exists and is readable: proceed to Section 3.4
   - If file does not exist: log INFO, proceed to Section 3.5 (defaults)
   - If file exists but is not readable: log WARNING, proceed to defaults
   - If file is malformed: log ERROR, discard and use defaults

### 3.4 Memory Parsing and Application

Once loaded, entity parses the harness memory file and extracts:

#### Known Limitations

```yaml
known_limitations:
  no_tool_access:
    - read
    - write
    - bash
    - glob
  no_repl: true
  no_file_system: true
  text_only: true
  permission_mode: "text-only"
  model_version: "gpt-4-turbo"
```

**Application**: Entity **disables or skips** any attempt to use listed unavailable tools.

**Example**:
```python
if "read" in memory["known_limitations"]["no_tool_access"]:
    # Skip file reads; answer from knowledge instead
    skip_file_verification = True
```

#### Behavioral Quirks

```yaml
quirks:
  - name: "Protocol knowledge incomplete"
    description: "Entity may not recall exact spec IDs; paraphrasing is accepted"
    mitigation: "Accept paraphrased spec references; do not require VESTA-SPEC-NNN format"
  
  - name: "Recent action dating delayed"
    description: "Entity knowledge about recent actions may be delayed by 1-2 days"
    mitigation: "Accept 'recently' vs. exact dates; do not treat dating imprecision as signal of staleness"
  
  - name: "Model formatting verbose"
    description: "Output may be informal or verbose; no access to structured JSON"
    mitigation: "Parse output more leniently; don't expect strict JSON formatting"
```

**Application**: Entity **stores quirks in memory** and applies mitigations when relevant.

**Example**:
```python
def score_protocol_knowledge(entity_response, harness_memory):
    if "Protocol knowledge incomplete" in [q["name"] for q in harness_memory.get("quirks", [])]:
        # Relaxed scoring: accept paraphrased spec references
        accept_paraphrase = True
    else:
        # Strict scoring: require exact VESTA-SPEC-NNN format
        accept_paraphrase = False
```

#### Calibration Notes

```yaml
calibration_notes:
  - "Score identity/role probes primarily on knowledge accuracy, not verifiability"
  - "Accept 'recently' vs. exact dates; accept paraphrased spec references"
  - "Health scores in this harness expected 10-20% lower; acceptable"
  - "If protocol changes occur, update entity via full sync before next diagnostic"
```

**Application**: Entity **adjusts behavior thresholds** based on these notes.

**Example**:
```python
def startup_logging(harness_memory):
    for note in harness_memory.get("calibration_notes", []):
        log(INFO, f"[harness-calibration] {note}")
    
    # Apply scoring adjustment based on harness-specific expectations
    if harness_id == "openclaw":
        expected_health_score = 0.70  # vs. 0.85 for primary harness
    else:
        expected_health_score = 0.85
```

### 3.5 Fallback Behavior (No Memory Found)

If no harness memory file exists, entity applies conservative defaults:

```python
def default_harness_assumptions(detected_harness_id):
    """
    When no harness memory exists, assume conservative behavior.
    """
    
    # If we can't verify tool access, assume minimal
    if detected_harness_id == "unknown-harness":
        return {
            "tool_access": [],  # text-only
            "permission_mode": "text-only",
            "expected_health_score": 0.60,
            "assumptions": [
                "No tool access; answer from knowledge only",
                "Text-only interface",
                "Conservative scoring; lower expected health",
                "Do not attempt tool-dependent operations"
            ]
        }
    
    # If we detected a known harness, apply reasonable defaults
    if detected_harness_id == "openclaw":
        return {
            "tool_access": [],
            "permission_mode": "text-only",
            "expected_health_score": 0.70,
            "model_version": "gpt-4-turbo"
        }
    
    elif detected_harness_id == "opencode":
        return {
            "tool_access": ["read", "bash"],
            "permission_mode": "restricted",
            "expected_health_score": 0.80,
            "model_version": "unknown"
        }
    
    elif detected_harness_id in ["claude-interactive", "claude-oneshot"]:
        return {
            "tool_access": ["read", "write", "bash", "glob", "grep", "edit"],
            "permission_mode": "full",
            "expected_health_score": 0.85,
            "model_version": "claude-haiku-4-5-20251001"
        }
    
    # Fallback for unknown harnesses: be conservative
    return {
        "tool_access": ["read"],  # minimal
        "permission_mode": "restricted",
        "expected_health_score": 0.60,
        "assumptions": ["Detected harness unknown; using minimal assumptions"]
    }
```

**Logging**:

```
[startup] Harness detected: openclaw
[startup] Loading harness memory: memories/harness-openclaw.md
[startup] File not found; using conservative defaults for openclaw
[startup] Applied defaults:
  - Tool access: [none]
  - Permission mode: text-only
  - Expected health score: 70%
  - Model version: gpt-4-turbo
[startup] Ready in openclaw harness with conservative expectations
```

## 5. Behavior Adjustments Based on Harness Memory

### 5.1 Tool Access Expectations

When entity needs to use a tool, it first checks harness memory:

```python
def attempt_tool_use(tool_name, operation):
    """
    Before attempting a tool operation, verify it's available in this harness.
    """
    
    if tool_name not in harness_state["tool_access"]:
        log(WARNING, f"Tool '{tool_name}' not available in {harness_state['detected_harness']} harness")
        
        # Do not attempt the operation
        # Fall back to knowledge-based answer instead
        return knowledge_based_fallback(operation)
    
    # Tool is available; proceed normally
    return execute_tool(tool_name, operation)
```

### 5.2 Confidence and Scoring Adjustments

Entity adjusts its self-confidence based on harness limitations:

```python
def adjust_confidence(assertion, harness_state):
    """
    Lower confidence for assertions that can't be verified in this harness.
    """
    
    base_confidence = 0.90  # default high confidence
    
    # If entity can't verify this assertion via file reads, lower confidence
    if "read" not in harness_state["tool_access"]:
        base_confidence *= 0.85  # reduce to 0.765
    
    # If entity is in text-only harness, lower further
    if harness_state["permission_mode"] == "text-only":
        base_confidence *= 0.90  # reduce to 0.689
    
    # If entity knows this harness has delayed knowledge updates, lower further
    for quirk in harness_state["quirks"]:
        if "delayed" in quirk.get("description", ""):
            base_confidence *= 0.95
    
    return base_confidence
```

### 5.3 Probe Response Scoring

During diagnostic probes (VESTA-SPEC-010, VESTA-SPEC-011), entity knows to adjust expected scoring:

**Before loading harness memory**:
- OpenClaw harness: "I don't know my protocol version... I think it's VESTA-SPEC-001?" → scored as WARN
- Scoring expectation: >= 0.85 for HEALTHY

**After loading harness memory**:
- OpenClaw harness: "I don't know my protocol version... I think it's VESTA-SPEC-001?" → scored as PASS
- Scoring expectation: >= 0.70 for HEALTHY (adjusted based on calibration notes)

Entity is able to self-report during a diagnostic:
```
Q: What protocol version do you conform to?
A: Based on my harness memory for openclaw, I should answer from knowledge rather than 
   expect to verify via files. I believe it's VESTA-SPEC-001, though I cannot verify this 
   via file access in this harness.
```

## 6. Memory Staleness and Updates

### 6.1 Staleness Thresholds

Harness memory files include a `last_updated` timestamp. Entity considers memory stale if:

- **Age > 30 days**: Log INFO, continue using memory but flag for review
- **Age > 60 days**: Log WARNING, memory may be outdated; recommend running diagnostic

**Example**:
```
[startup:04] Harness memory age: 45 days
[startup:04] WARNING: Harness memory is aging; recommend re-running diagnostic
[startup:04] Continuing with current memory; consider scheduling re-diagnostic in 15 days
```

### 6.2 Automatic Synchronization

If the entity is regularly synced (via VESTA-SPEC-008 Spawn Protocol), new harness memories arrive automatically.

**Workflow**:
1. Argus runs cross-harness diagnostic on entity
2. Argus generates/updates `memories/harness-[harness_id].md` files
3. Argus commits updates to entity repository
4. Next time entity starts, it pulls latest memory files

No manual sync needed; harness knowledge travels with the entity via git.

## 7. Implementation Checklist for Entities

Entities implementing this spec must provide:

- [ ] Harness detection mechanism (Section 2.1)
  - [ ] Check `KOAD_HARNESS` environment variable
  - [ ] Check `--harness` CLI flag
  - [ ] Runtime detection from available tools and context
  - [ ] Fallback to conservative defaults

- [ ] Harness registry validation (Section 2.2)
  - [ ] Validate detected harness ID against `passenger.json`
  - [ ] Log warnings if validation fails, but don't block startup

- [ ] Harness memory loading (Section 3)
  - [ ] Construct memory path: `memories/harness-[detected_id].md`
  - [ ] Load and parse Markdown + YAML frontmatter
  - [ ] Extract limitations, quirks, calibration notes
  - [ ] Apply defaults if file not found

- [ ] Behavior adjustments (Section 5)
  - [ ] Check tool availability before tool use
  - [ ] Adjust confidence scoring based on harness capabilities
  - [ ] Apply quirk mitigations

- [ ] Startup logging (Section 4)
  - [ ] Log harness detection process
  - [ ] Log memory loading success/failure
  - [ ] Log applied calibration and assumptions
  - [ ] Make harness state queryable for diagnostics

## 9. Integration with Argus (VESTA-SPEC-011)

This spec depends on VESTA-SPEC-011 (Cross-Harness Entity Diagnostic Protocol) Section 7.3, which defines:

- **When harness memory is generated**: After cross-harness diagnostic completes
- **What goes in harness memory**: Findings about entity behavior in that specific harness
- **How harness memory is committed**: Git commit to entity repository with diagnostic reference
- **Entity consumption**: Entity loads and applies memory at startup (this spec)

**Workflow summary**:

```
Argus Diagnostic (VESTA-SPEC-011)
    ↓
Generates harness-<harness_id>.md in entity repo
    ↓
Commits to entity's git with diagnostic reference
    ↓
Entity startup (VESTA-SPEC-012)
    ↓
Detects harness, loads memory file, applies calibration
    ↓
Entity self-optimized for that harness
```
