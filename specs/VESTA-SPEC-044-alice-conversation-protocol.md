# VESTA-SPEC-044: Alice Conversation Protocol

**Authority:** Vesta (platform stewardship). This spec defines the runtime model for an Alice curriculum conversation — what happens from session open to completion write, what state lives on disk, and how Vulcan reads that state to gate progression.

**Scope:** Session lifecycle, session state format, completion write contract, accelerated assessment path, and the boundary between what Alice owns (writes) and what Vulcan owns (reads).

**Consumers:**
- Alice — the entity that conducts conversations and writes all state
- Vulcan — reads completion records and identity files to drive PWA gating
- Chiron — authors curriculum content; observes session feedback to revise content
- Argus — audits completion record conformance

**Status:** Canonical. Derived from Muse brief (alice-ui-design-brief.md, 2026-04-05 update) and Chiron DECISIONS.md.

---

## 1. Boundary of Authority

This spec is built around a single invariant:

> **Alice writes. Vulcan reads. Vulcan never writes learner state.**

The filesystem is the source of truth. Alice is its only writer. Vulcan polls it to determine what the PWA should display.

| Actor | Reads | Writes |
|-------|-------|--------|
| Alice | Curriculum content from `~/.chiron/curricula/` | All files under `~/.alice/learners/{uuid}/` |
| Vulcan | All files under `~/.alice/learners/{uuid}/` | Nothing in the learner tree |
| Chiron | Learner feedback signals (aggregated) | Curriculum content in `~/.chiron/curricula/` |

---

## 2. Learner Identity

A learner identity is established on first interaction. It is never derived from an external account. It lives on disk.

### 2.1 Identity File

**Path:** `~/.alice/learners/{uuid}/identity.md`

```markdown
---
learner_id: a9f3c2e1-7b4d-4f2a-9c8e-123456789abc
display_name: Jordan
created_at: 2026-04-05T14:22:00Z
last_seen_at: 2026-04-05T16:08:41Z
---
```

**Fields:**

| Field | Type | Semantics |
|-------|------|-----------|
| `learner_id` | UUIDv4 | Generated by Alice at first interaction. Never shown in normal flow. Used as directory key and on the graduation certificate. |
| `display_name` | string | Learner's chosen name or handle. Alice uses this in all conversation. Shown on certificate. |
| `created_at` | ISO-8601 | Timestamp of learner creation (name was confirmed). |
| `last_seen_at` | ISO-8601 | Updated by Alice at the start of each new session. Used for resumption context. |

**Write invariants:**
- `learner_id` is written once and never changed.
- `display_name` may be updated if learner requests a name change mid-curriculum. Alice confirms before overwriting.
- `created_at` is written once and never changed.

### 2.2 UUID Generation

Alice generates a UUIDv4 (random) locally when the learner confirms their display name. No external service. No user account needed. The UUID is the address of everything Alice knows about this learner.

## 3. Session Lifecycle

A session is a contiguous interaction between one learner and Alice, from greeting to suspension or completion.

### 3.1 Session Phases

```
1. OPEN       — Alice identifies learner (new or returning)
2. CONTEXT    — Alice loads learner's current level and progress
3. ACTIVE     — Conversation proceeds (Alice teaches, learner responds)
4. CHECKPOINT — Alice evaluates whether exit criteria are met
5. WRITE      — If exit criteria pass, Alice writes completion record
6. ADVANCE    — Next level becomes available; session continues or suspends
7. SUSPEND    — Session ends without completion (learner stops, timeout, etc.)
```

### 3.2 Session State File

Alice maintains a live session state file while a conversation is active. This file is overwritten on each state transition. It is NOT the completion record — it is transient state that enables resumption.

**Path:** `~/.alice/learners/{uuid}/curricula/{curriculum_id}/session-state.md`

```markdown
---
learner_id: a9f3c2e1-7b4d-4f2a-9c8e-123456789abc
curriculum_id: alice-onboarding
current_level: 3
session_id: 7f2e9a1b-0c3d-4e5f-a6b7-c8d9e0f1a2b3
session_opened_at: 2026-04-05T14:22:00Z
session_updated_at: 2026-04-05T14:31:17Z
phase: active
message_count: 12
---

# Session Context

Level 3: Keys & Identity — in progress.

Learner has acknowledged the concept of key-as-identity. Currently discussing
the distinction between data ownership and key ownership. Has asked two
clarifying questions; both answered.

Next checkpoint: Alice will present the "your keys are you" summary and ask
learner to restate the principle in their own words.
```

**Fields:**

| Field | Type | Semantics |
|-------|------|-----------|
| `learner_id` | UUID | Back-reference to identity |
| `curriculum_id` | string | Identifies the curriculum (e.g. `alice-onboarding`) |
| `current_level` | integer | Level number currently active (0-based) |
| `session_id` | UUIDv4 | Fresh UUID per session open. Not persisted long-term — used for debugging and audit only. |
| `session_opened_at` | ISO-8601 | When this session opened |
| `session_updated_at` | ISO-8601 | Last Alice write (Alice updates this every 2–3 exchanges) |
| `phase` | enum | One of: `opening`, `active`, `checkpoint`, `writing`, `suspended` |
| `message_count` | integer | Running count of messages in this session |

The prose body is Alice's working context: a running summary of what has been covered, where the learner seems to be, and what Alice intends to do next. This is not a log — it is Alice's working memory for the session. It intentionally does not record the full conversation (that would be too large). It records understanding, not utterances.

**Lifecycle:**
- Created or overwritten at session open.
- Updated by Alice at each meaningful transition (not every message).
- Deleted after a successful completion write (the completion record supersedes it).
- Retained on suspension — Alice reads it on next session open to resume context.

## 4. Completion Write Contract

### 4.1 When Alice Writes

Alice writes a completion record when **all** of the following hold:

1. The learner has reached the end of the level's content.
2. The exit criteria for the level are satisfied (per the curriculum bubble's `exit_criteria` field; see VESTA-SPEC-025 §5).
3. The learner has not expressed confusion about the core concept being assessed.

Alice does not write on a timer. Alice does not write on message count. Alice writes when understanding is demonstrated. This is a judgment call Alice makes in the moment of the conversation.

**Premature write risk:** If Alice writes before understanding is demonstrated, the curriculum is undermined. Vulcan will gate based on the presence of the file. The file's presence means "this level is complete." Alice is the only guardian of what that means.

### 4.2 Completion Record Format

**Path:** `~/.alice/learners/{uuid}/curricula/{curriculum_id}/level-{N}-complete.md`

**Example:** `~/.alice/learners/a9f3c2e1.../curricula/alice-onboarding/level-3-complete.md`

```markdown
---
learner_id: a9f3c2e1-7b4d-4f2a-9c8e-123456789abc
curriculum_id: alice-onboarding
level: 3
level_title: "Keys & Identity"
completed_at: 2026-04-05T14:44:00Z
duration_seconds: 1320
exit_criteria_met:
  - "Learner restated the key-as-identity principle in own words"
  - "Learner distinguished between data ownership and key ownership"
accelerated: false
---

# Level 3 Completion Record

Jordan completed Level 3 (Keys & Identity) after a 22-minute conversation.
Jordan demonstrated understanding by explaining that "your keys are you, not
your data" and described the practical difference between knowing a password
and holding a private key.

Alice assessed: exit criteria satisfied. Completion record written at 14:44 UTC.
```

**Fields:**

| Field | Type | Semantics |
|-------|------|-----------|
| `learner_id` | UUID | Links back to identity |
| `curriculum_id` | string | Curriculum identifier |
| `level` | integer | Level number (matches filename) |
| `level_title` | string | Human-readable level name |
| `completed_at` | ISO-8601 | Timestamp of Alice's write |
| `duration_seconds` | integer | Elapsed seconds from session open to completion write for this level. Approximate — derived from session state timestamps. |
| `exit_criteria_met` | string[] | List of exit criteria as Alice assessed them (drawn from curriculum bubble). One line per criterion. Verbatim or paraphrased from `exit_criteria` in SPEC-025. |
| `accelerated` | boolean | `true` if this completion was via accelerated assessment path (see §4.3). `false` for normal progression. |

**Prose body:** Alice's narrative of how the level was completed. 2–4 sentences. Not a transcript. Written by Alice in natural language. Consumed by Chiron as feedback signal.

### 4.3 Accelerated Assessment Path

When a learner claims prior knowledge of a locked level and requests assessment (per the UI affordance in the Muse brief), the path is:

1. Alice enters the conversation in assessment mode for the claimed level.
2. Alice presents the level's exit criteria as a structured assessment.
3. Learner responds.
4. If Alice judges exit criteria satisfied: Alice writes the completion record with `accelerated: true`.
5. Vulcan reads the completion record. The presence of the file is sufficient — Vulcan does not inspect the `accelerated` field for gating. Gating is file-presence only.

The `accelerated` field is for Chiron's analytics and curriculum quality feedback, not for access control.

### 4.4 What Vulcan Reads

Vulcan determines level status by checking file presence only:

```
level status = "complete"   if  level-{N}-complete.md exists AND is well-formed
             = "available"  if  level-{N-1}-complete.md exists (or N=0)
             = "locked"     otherwise
```

**Well-formed:** YAML frontmatter is parseable, `learner_id` matches current session UUID, `level` matches filename level number. If malformed: log an error, treat as `locked`, do not overwrite the file.

## 5. Certificate Write

When level 12 is complete, Alice writes the graduation certificate. This is described fully in `alice-graduation-certificate-protocol.md`. A brief summary for completeness:

- Alice writes `~/.alice/learners/{uuid}/curricula/alice-onboarding/certificate.md`
- Alice signs with her Ed25519 key (`~/.alice/id/ed25519`)
- The certificate includes `learner_id`, `display_name`, `completed_at`, and the Ed25519 signature
- Vulcan reads this file to render the graduation screen (Screen 4 in the Muse brief)
- Vulcan does not write or sign the certificate

## 6. Session Resumption

When a learner returns after a previous session was suspended:

1. Alice locates the learner's identity by UUID (or asks "Have we spoken before?" if the device is new — see §7 for cross-device constraints).
2. Alice reads `session-state.md` for the current level.
3. Alice reads the prose body of `session-state.md` to reconstruct working context.
4. Alice resumes the conversation with a brief orientation: "When we left off, we were discussing [topic]. Ready to continue?"
5. Alice does not re-read the full conversation history. The session state prose is the context handoff. Alice authored it for herself at suspension time.

**Suspension write:** Before Alice's process terminates, Alice writes a final `session-state.md` update with `phase: suspended` and an updated prose summary. This is the graceful shutdown path. If Alice terminates ungracefully (kill signal, crash), the last-written `session-state.md` is still usable — it will be slightly stale but recoverable.

## 7. Multi-Device Constraint

This spec acknowledges the multi-device problem but does not solve it. Vesta documents the constraint here as a decision boundary, not a gap to be filled in this release.

**The constraint:** All learner state (`~/.alice/learners/{uuid}/`) is local to the machine where Alice runs. If a learner switches devices (phone to laptop, home to office), Alice on the new device has no record of them. The learner appears new.

**Recovery paths available today (without new infrastructure):**
1. The learner provides their UUID (shown on the graduation certificate, or retrievable from their first device). Alice creates a stub identity file with the provided UUID and `display_name`. Prior completion records are lost — the learner must re-demonstrate mastery.
2. The learner manually copies `~/.alice/learners/{uuid}/` to the new device. This is technically correct but user-hostile.

**What is not yet designed:**
- Sync protocol for learner state across devices
- Cloud backup of `~/.alice/` (this would require a sync backend, introducing a sovereignty tradeoff)
- UUID-based recovery UX ("I have my learner ID from another device")

**Constraint status:** Accepted for Phase 1. Alice is single-device. The graduation certificate carries the UUID as the learner's portable credential. Cross-device sync is deferred to a separate spec (VESTA-SPEC-047).

**No Vulcan action needed on this constraint.** Vulcan implements against the local filesystem model. If a sync layer is added later, it will be additive — the local filesystem model remains primary.

## 8. Directory Structure Summary

```
~/.alice/
  learners/
    {uuid}/
      identity.md                                      ← written once on first interaction
      curricula/
        alice-onboarding/
          session-state.md                             ← live, overwritten; deleted after completion
          level-0-complete.md                          ← written by Alice when level passes
          level-1-complete.md
          ...
          level-11-complete.md
          certificate.md                               ← written by Alice on Level 12 completion
```

All files are markdown with YAML frontmatter. No binary files. No database. The directory tree is the state.
